## 2.1 전유미 구조 유지 + GitHub(전상표) push·연계 시 꼬임 검토

**결정:** 이 PC는 **전유미 사용자 구조로 작업을 계속**하는 것이, 지금까지 진행한 작업을 안전하게 보존하는 데 유리함.  
다만 OneDrive가 **전상표** 계정이고, GitHub 저장소 소유자도 **전상표(spjun21@gmail.com)** 이므로, **폴더·개발 파일 저장 위치**와 **push·연계 작업 방식**에 따라 꼬일 수 있는 구간을 아래와 같이 정리함.

### 2.1.1 꼬임이 날 수 있는 경우

| 구간 | 설명 | 위험 |
|------|------|------|
| **개발 폴더를 전상표 OneDrive 안에 둔 상태에서 push** | 프로젝트가 `OneDrive\상표 - 개인\...` 같은 **전상표 OneDrive 동기화 폴더 안**에 있음. 여기서 `git push` 하면 OneDrive가 같은 폴더를 동시에 동기화하면서 **파일 잠금·충돌·경로 변경**이 발생할 수 있음. | 동기화 중인 파일이 Git에 의해 수정되면 OneDrive가 덮어쓰거나, 반대로 OneDrive 동기화가 `.git` 또는 소스 파일을 꼬이게 할 수 있음. |
| **OneDrive에 저장해 두었던 개발 파일과 GitHub 연계** | 예: OneDrive(전상표) 폴더에 있던 파일을 수정한 뒤 같은 내용을 GitHub(전상표)에 push 하려 할 때. **작업 디렉터리가 OneDrive 안**이면 동기화와 Git이 같은 경로를 동시에 사용해 충돌 가능. | 편집 중 동기화 지연·충돌로 **저장 내용이 어긋나거나**, push 시점의 파일 상태가 예상과 다를 수 있음. |
| **여러 PC·OneDrive·Git 동시 사용** | 전상표 OneDrive 폴더 안의 프로젝트를 다른 PC에서도 열어 수정하고, 이 PC(전유미)에서도 수정 후 push 하면, OneDrive 동기화와 Git 히스토리가 서로 맞지 않아 **어느 쪽이 최신인지 혼란** 가능. | 중복 편집·동기화 순서에 따른 덮어쓰기. |

**정리:** 꼬임의 직접 원인은 “GitHub가 전상표 소유”인 것이 아니라, **프로젝트·작업 폴더를 전상표 OneDrive 동기화 폴더 안에 두고** 그 안에서 Git(push/pull)과 OneDrive를 동시에 쓰는 경우임.

### 2.1.2 꼬임이 나지 않는 경우 (현재 SPK Mobile Bot 구조)

| 항목 | 설명 |
|------|------|
| **현재 프로젝트 경로** | `C:\Users\user\.cursor\spk  mobile  bot\spk-mobile-bot` → **전유미 사용자 폴더** 안이며, **전상표 OneDrive 폴더 밖** (OneDrive 기본 동기화 경로가 아님). |
| **push 동작** | `git push`는 **로컬 → GitHub 서버**로만 가므로, **저장 위치가 OneDrive 밖**이면 OneDrive와 겹치지 않음. 전유미 PC에서 전상표(spjun21) GitHub 계정으로 push 하는 것 자체는 **계정 불일치로 인한 기술적 꼬임 없음** (Git 인증만 전상표 계정으로 되어 있으면 됨). |
| **연계 작업** | 작업 디렉터리를 **OneDrive 밖**(지금처럼 `.cursor` 아래)에 두고, 여기서 편집·commit·push 하면 OneDrive 동기화와 Git이 **같은 폴더를 두고 경쟁하지 않아** 꼬임 가능성이 낮음. |

### 2.1.3 전유미 구조 유지 시 권장 사항 (작업 보존 + push·연계 안전)

1. **프로젝트·개발 파일 위치**
   - **SPK Mobile Bot** 및 다른 GitHub 연계 프로젝트는 **전상표 OneDrive 폴더 안에 두지 말고**, **전유미 사용자 폴더 안**(예: `.cursor` 아래 또는 `C:\Users\user\` 의 다른 로컬 경로)에만 두기.
   - 과거에 **전상표 OneDrive 폴더에 저장해 두었던 개발 파일·폴더**가 있다면, **연계·push 할 때는** OneDrive 밖(전유미 로컬)으로 **복사한 뒤** 그 복사본을 작업 디렉터리로 사용하고, push는 그 위치에서만 수행하기.

2. **push·연계 작업**
   - 작업은 항상 **OneDrive 밖**의 로컬 폴더에서 진행.
   - GitHub(전상표 spjun21)에 push/풀은 **지금처럼** 해당 로컬 저장소에서 실행. (인증만 전상표 계정으로 설정되어 있으면 됨.)
   - OneDrive(전상표)에 있는 자료는 **참고·복사용**으로만 쓰고, **실제 편집·Git 작업은 로컬 복사본**에서만 하기.

3. **정리**
   - **전유미 사용자 구조 유지** → 기존 작업 경로·환경이 그대로라 **작업 보존에 유리**.
   - **개발·push·연계는 전상표 OneDrive 폴더 밖에서만** 하면, 전상표 OneDrive 계정과 GitHub(전상표)를 함께 써도 **꼬임 가능성을 줄일 수 있음**.

## 3. OneDrive로 인한 경로 변경 우려

### 3.1 경로가 바뀌는 경우 (일반적)
- OneDrive **“데스크톱, 문서, 사진 백업”** 사용 시  
  → `C:\Users\user\Desktop`, `문서`, `사진` 등이  
  `C:\Users\user\OneDrive\Desktop` 등으로 **이동**되며 경로가 변경됨.
- 현재 프로젝트는 **`.cursor` 아래**에 있으므로,  
  위 “중요 폴더 백업”만 켜져 있다면 **이동 대상이 아님**.

### 3.2 그래도 주의할 점
- **프로젝트를 Desktop/문서/사진 안으로 옮긴 경우**  
  → 해당 폴더가 OneDrive로 이동하면 프로젝트 경로도 함께 바뀜.
- **OneDrive가 사용자 폴더 전체를 대체하는 정책/설정**이 있는 환경  
  → 드물지만, 이론상 사용자 경로 구조가 바뀔 수 있음.
- **동기화/잠금**  
  → OneDrive 동기화 중 파일 잠금·충돌로 Git/빌드 작업에 간헐적 문제가 생길 수 있음.

## 4. 로컬 저장 경로 안정적 유지를 위한 OneDrive 조절 권장사항

### 4.1 프로젝트 위치 유지 (권장)
- **지금처럼 프로젝트를 `C:\Users\user\.cursor\...` 아래에 두는 것을 유지**하는 것이 좋음.  
  → OneDrive “데스크톱, 문서, 사진 백업”의 기본 이동 대상이 아니어서 경로가 안정적.

### 4.2 OneDrive “중요 폴더 백업” 사용 시
- **설정 경로:**  
  작업 표시줄 OneDrive 아이콘 → 설정(톱니바퀴) → **동기화 및 백업** → **내 중요한 Windows 폴더 백업 관리**
- **선택지:**
  - **백업 사용:** Desktop/문서/사진이 OneDrive 폴더로 이동.  
    → **프로젝트는 Desktop·문서·사진 밖에 두기** (현재처럼 `.cursor` 아래 유지).
  - **백업 미사용:** 이 폴더들은 로컬 `C:\Users\user\` 아래에만 있음.  
    → 경로 변경 가능성은 더 적음.

### 4.3 프로젝트 폴더를 OneDrive 동기화에서 제외 (선택)
- 프로젝트가 **어쩌다 OneDrive 폴더 안**에 있다면:
  - **방법 1:** 프로젝트를 **OneDrive 밖**으로 이동  
    (예: `C:\Users\user\.cursor\spk  mobile  bot\spk-mobile-bot` 같은 경로).
  - **방법 2:** OneDrive 설정에서 해당 폴더를 **동기화 제외**  
    (설정 → 계정 → “선택한 폴더 동기화” 등에서 제외).  
  → Git, node_modules, 빌드 산출물 등이 동기화되지 않아 충돌·경로 이슈를 줄일 수 있음.

### 4.4 이미 백업을 켜 둔 상태라면
- **Desktop/문서/사진만** 이동되는지 확인.
- **개발 프로젝트는 이 세 폴더 밖**에 두었는지 확인.
- 필요하면 프로젝트를 **지금 경로(`.cursor` 아래)로 이동**해 두는 것을 권장.

### 4.5 경로 변경 시 대비 (Git 활용)
- **Git + GitHub**를 쓰고 있으므로:
  - 로컬 경로가 바뀌어도 **같은 폴더를 새 위치에 다시 clone**하면 복구 가능.
  - 중요한 작업은 **자주 `git push`** 해 두면, 경로 변경·PC 변경 시에도 안전.
